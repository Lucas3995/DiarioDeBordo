**ADR-001 — `docs/adr/ADR-001-ui-framework.md`:**
- Título: "Framework de Interface — Avalonia UI + SukiUI"
- Contexto: sistema desktop nativo cross-platform (Linux + Windows) sem dependência de browser; necessidade de tema escuro, componentes customizáveis, WCAG 2.2 AAA
- Decisão: "Adotamos Avalonia UI como framework de UI e SukiUI como biblioteca de componentes. O padrão MVVM é implementado com CommunityToolkit.Mvvm."
- Consequências positivas: runtime próprio (sem WebView), cross-platform (Linux + Windows), suporte a MVVM nativo, SukiUI fornece temas claro/escuro prontos
- Consequências negativas: ecossistema menor que WPF; SukiUI tem menos componentes que MUI/Ant Design
- Alternativas: MAUI (não suporta Linux desktop); Electron (não é nativo, consome mais memória); WPF (somente Windows)
- Citar: Padrões Técnicos v4, seção 2.1

**ADR-002 — `docs/adr/ADR-002-banco-de-dados.md`:**
- Título: "Banco de Dados — PostgreSQL bundled na porta 15432"
- Contexto: aplicação desktop offline-first; usuário não deve instalar ou configurar banco de dados manualmente; PostgreSQL escolhido pela robustez e suporte a full-text search nativo
- Decisão: "Adotamos PostgreSQL como banco de dados, distribuído bundled com o instalador da aplicação, rodando na porta 15432 (não-padrão para evitar conflito com instalações existentes do usuário). A senha é gerada na instalação e armazenada no Secure Storage do OS."
- Consequências positivas: zero configuração manual, senha forte por padrão, porta não-padrão reduz colisão, full-text search nativo
- Consequências negativas: aumenta tamanho do instalador (~80MB); requer bundling do runtime PostgreSQL por plataforma
- Alternativas: SQLite (sem full-text search adequado, sem multi-user local); SQL Server Express (somente Windows)
- Citar: Padrões Técnicos v4, seção 3.1

**ADR-003 — `docs/adr/ADR-003-arquitetura.md`:**
- Título: "Arquitetura — Monolito Modular com Bounded Contexts"
- Contexto: sistema desktop com 8 bounded contexts; microserviços têm overhead desnecessário para desktop; mas separação por contexto é necessária para manutenibilidade e fases incrementais
- Decisão: "Adotamos arquitetura de monolito modular onde cada bounded context é um projeto/namespace separado (`DiarioDeBordo.Module.Acervo`, etc.) com interfaces explícitas definidas em `DiarioDeBordo.Core`. Comunicação entre módulos exclusivamente via MediatR (Commands, Notifications) ou interfaces em Core — sem referência direta entre módulos."
- Consequências positivas: deploy único, comunicação in-process via MediatR (sem rede), separação explícita de responsabilidades, facilidade de teste por módulo
- Consequências negativas: acoplamento de deploy (todos os módulos sobem juntos); migração para microserviços no futuro exigiria extração de módulos
- Alternativas: microserviços (overhead de rede desnecessário para desktop); monolito sem modularização (dificulta evolução por fase)
- Citar: Padrões Técnicos v4, seção 1.2 e seção 6

**ADR-004 — `docs/adr/ADR-004-stack-tecnologica.md`:**
- Título: "Stack Tecnológica — C#/.NET 9 LTS + MediatR + EF Core + Velopack"
- Contexto: necessidade de stack madura, cross-platform, com bom suporte a DDD e CQRS; instalador que não exige .NET pré-instalado
- Decisão: "Adotamos C# com .NET 9 LTS. MediatR para comunicação interna entre módulos (CQRS + mediator pattern). EF Core com provider PostgreSQL para persistência. Velopack para distribuição e auto-update com code signing."
- Consequências positivas: .NET 9 LTS tem suporte de longo prazo; MediatR permite desacoplamento entre módulos; EF Core + PostgreSQL é combinação madura; Velopack lida com diferenças de instalação Linux/Windows
- Consequências negativas: MediatR pode esconder dependências implícitas se mal usado; EF Core adiciona overhead de ORM
- Alternativas: Dapper (menos abstração, mais SQL manual); NServiceBus (overkill para desktop); NSIS installer (mais manual que Velopack)
- Citar: Padrões Técnicos v4, seção 2

**ADR-005 — `docs/adr/ADR-005-seguranca.md`:**
- Título: "Abordagem de Segurança — Argon2id, DPAPI/libsecret, BannedSymbols"
- Contexto: aplicação desktop com dados pessoais do usuário; autenticação local sem OAuth; credenciais do banco precisam ser protegidas; necessidade de prevenir regressões de segurança via análise estática
- Decisão: "Adotamos: (1) Argon2id para hash de senhas de usuário; (2) DPAPI (Windows) e libsecret (Linux) para armazenar credenciais do banco PostgreSQL — nunca em arquivos de configuração; (3) BannedSymbols.txt para impedir uso de APIs perigosas (string para senhas, SQL concatenado, DTD processing); (4) `CryptographicOperations.ZeroMemory()` após uso de dados sensíveis em memória; (5) `usuarioId` obrigatório em toda query ao banco."
- Consequências positivas: senhas nunca em plaintext; credenciais do banco protegidas pelo OS; análise estática impede regressões; ZeroMemory reduz janela de exposição
- Consequências negativas: Argon2id é mais lento que bcrypt por design (feature, não bug); libsecret requer D-Bus no Linux (disponível em todos os desktops Linux modernos)
- Alternativas: bcrypt (menos resistente a GPUs que Argon2id); config files encriptados (chave de encriptação precisa ficar em algum lugar); sem BannedSymbols (depende de revisão manual)
- Citar: Padrões Técnicos v4, seção 4
