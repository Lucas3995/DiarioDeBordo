# Phase 3 — UI Design Contract

> Visual and interaction contract for Acervo Básico. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> 
> **Scope:** CRUD completo de conteúdo com modal de detalhe/edição, categorias como chips, relações bidirecionais, histórico de consumo como timeline de sessões, classificação e nota independentes.

## Modal Layout — Scientific Foundation (D-05)

### Research Basis

The modal layout is grounded in peer-reviewed literature on complex form organization, progressive disclosure, and personal information management in desktop interfaces:

1. **Miller, G. A. (1956).** "The magical number seven, plus or minus two: Some limits on our capacity for processing information." *Psychological Review, 63*(2), 81–97. → Sections limited to 5–7 visible fields maximum. Groups formed by cognitive proximity.

2. **Shneiderman, B. & Plaisant, C. (2010).** *Designing the User Interface: Strategies for Effective Human-Computer Interaction.* 5th ed. Addison-Wesley. → Progressive disclosure: show essential fields first; reveal optional fields on demand. Already established in the project (CriarConteudoView.axaml cites Shneiderman 1996).

3. **Norman, D. A. (2013).** *The Design of Everyday Things.* Revised & expanded ed. Basic Books. → Visibility of system state: unsaved changes indicator. Constraints that prevent errors: confirmation before destructive actions. Already referenced in AcervoView.axaml empty state.

4. **Wroblewski, L. (2008).** *Web Form Design: Filling in the Blanks.* Rosenfeld Media. → Top-aligned labels complete 26% faster than left-aligned (Matteo Penzo eye-tracking study, cited in ch. 5). Logical grouping reduces form abandonment. Though web-focused, principles validated across modalities.

5. **Cooper, A., Reimann, R. & Cronin, D. (2014).** *About Face: The Essentials of Interaction Design.* 4th ed. Wiley. → Sovereign posture applications (used frequently, complex data) benefit from modeless in-context editing with progressive reveal and persistent workspace state.

6. **Galitz, W. O. (2007).** *The Essential Guide to User Interface Design.* 3rd ed. Wiley. → Single-page scrollable forms with clear section headers are completed faster than tabbed forms for 4–8 sections (ch. 17). Tabs hide content and add navigation overhead for forms where sections are interdependent.

7. **Jones, W. & Teevan, J. (2007).** *Personal Information Management.* University of Washington Press. → Users managing personal collections prefer flexible categorization (tags over hierarchies), incremental annotation, and low-friction recording that doesn't interrupt the primary task.

### Layout Decision: Sectioned Scrollable Form with Collapsible Sections

**Chosen pattern:** Single-column scrollable modal with 4 collapsible sections (accordion-style `Expander` controls). NOT tabs.

**Rationale:**
- Galitz (2007): scrollable > tabs for ≤8 interdependent sections — no hidden state
- Shneiderman (2010): progressive disclosure via collapse — essential section expanded, optional sections collapsed
- Miller (1956): 4 sections with 3–6 fields each stays within working memory limits
- Cooper (2014): sovereign posture — persistent form state visible at a glance via section headers

### Section Order (cognitive flow)

| # | Section | Default State | Fields (visible when expanded) |
|---|---------|---------------|-------------------------------|
| 1 | Identificação | **Expanded** | Título*, Descrição, Anotações, Formato, Subtipo |
| 2 | Avaliação | Collapsed | Nota [0-10], Classificação (Gostei/NãoGostei/Nulo), Progresso (estado + posição) |
| 3 | Organização | Collapsed | Categorias (chips + autocomplete), Relações (lista + adicionar) |
| 4 | Histórico de Consumo | Collapsed | Timeline de sessões, Total esperado, Progresso calculado, botão "Registrar sessão" |

*Título is the only required field (I-01). All others are optional — consistent with progressive disclosure and ACE-02 "incrementally add".

**Section headers** display summary metadata when collapsed:
- Identificação: always shows Título in header even when collapsed (identification at a glance)
- Avaliação: shows "Nota: 8.5 · 👍" or "Sem avaliação" when collapsed
- Organização: shows "3 categorias · 2 relações" or "Sem categorias ou relações" when collapsed
- Histórico: shows "12 de 24 (50%)" or "4 sessões registradas" or "Sem sessões" when collapsed

**Layout:** `NumericUpDown` or `Slider` + `NumericUpDown` combo.

- FormatString: "0.#" (show one decimal only if non-zero)
- Width: 100px
- Watermark: "—" when null (not rated)
- Clear button to set back to null
- `AutomationProperties.Name="Nota pessoal de 0 a 10"`

## Interaction Contracts

### Card → Modal Flow (D-01, D-04)

1. User clicks "Ver detalhe" icon button on card
2. `AcervoViewModel` creates `ConteudoDetalheViewModel` via factory
3. `ConteudoDetalheModal` opens as modal dialog (`ShowDialog`)
4. Modal loads content detail via `ObterConteudoDetalheQuery` (categories, relations, sessions)
5. All sections initially collapsed except Identificação
6. User edits fields → ViewModel tracks dirty state (`IsDirty` property)
7. On "Salvar" → validate → `EditarConteudoCommand` → close modal → refresh list
8. On "Cancelar" or X → if `IsDirty`, show discard confirmation dialog → close or return
9. On "Excluir" → show delete confirmation dialog → `RemoverConteudoCommand` → close modal → refresh list

### Category Autocomplete Flow (D-09, D-10)

1. User types in category input field
2. After 200ms debounce and ≥1 character: query `ListarComAutocompletarAsync`
3. Popup shows matching categories (max 8)
4. If no exact match: append "Criar categoria '{texto}'" as last item
5. User selects existing → chip added, input cleared
6. User selects "Criar" → `ObterOuCriarAsync` → chip added, input cleared
7. Chips render with Origem distinction (D-11): manual = default style, automática = accent tint + asterisco
8. User clicks X on chip → category removed from content (no confirmation needed — not destructive, can re-add)

### Relation Creation Flow (D-12, D-14, D-15)

1. User clicks "Adicionar relação" in Organização section
2. Inline form expands: content search + type selector
3. User searches for target content → autocomplete with acervo items (excludes current content and already-related content)
4. User selects target → target title displayed as confirmation
5. User selects or creates relation type
6. If creating new type: secondary field for inverse name appears
7. User clicks "Vincular" → `CriarRelacaoCommand` → creates both A→B and B→A
8. New relation appears in list immediately
9. Inline form stays open for adding more relations

### Session Registration Flow (D-20, D-21)

1. User clicks "Registrar sessão" in Histórico section
2. Inline miniform expands with Título/Posição + Data (today default)
3. User fills required fields
4. Optional: clicks "Mais detalhes" to reveal Anotações, Nota, Classificação
5. Clicks "Registrar" → `RegistrarSessaoCommand` → new child Conteudo created with "Contém"/"Parte de" relation
6. Session appears in timeline above
7. Form clears but stays open for next entry
8. Progress recalculated automatically
9. User clicks "Fechar" or collapses Histórico section to dismiss form

### Unsaved Changes Guard (D-02)

- `IsDirty` tracked in ViewModel: compares current field values against loaded snapshot
- Visual indicator: modal title suffix " •" (bullet) when dirty, or asterisk in window title
- Triggered on: Cancel button, X close button, Escape key
- Dialog: "Descartar alterações?" with "Descartar" (destructive) + "Voltar ao formulário" (default)

### Keyboard Navigation (WCAG 2.1.1, 2.4.3)

- Tab order follows visual order: section expanders → fields within sections → footer buttons
- Escape: close modal (with dirty check)
- Enter on Expander header: toggle expand/collapse
- Enter on autocomplete item: select it
- Arrow keys in autocomplete popup: navigate suggestions
- Delete/Backspace in empty category input: focus last chip for removal

### Card Delete Flow (D-03)

1. User clicks delete icon on card
2. Confirmation dialog opens (same as modal delete)
3. On confirm → `RemoverConteudoCommand` → card removed from list → status bar updated
4. On cancel → dialog closes, no action
